# AI 的真实缺陷

**在拥抱 AI 的同时，必须清醒认识它的不足：**

| 缺陷 | 说明 | 影响 |
|------|------|------|
| **幻觉（Hallucination）** | 一本正经地说错话，而且非常自信 | 不能盲目信任 AI 的输出 |
| **成本（Cost）** | GPT-4/Claude 级别的 API 大规模使用很贵 | 需要精打细算 Token 用量 |
| **隐私（Privacy）** | 数据发送到第三方服务器 | 敏感业务数据不能直接喂给 AI |
| **不可解释（Explainability）** | 无法解释"为什么"给出这个答案 | 在合规要求高的场景受限 |
| **过度依赖（Over-reliance）** | 团队基本功可能退化 | 需要保持自主编码和思考能力 |

<div class="callout">

**重要提醒：** AI 是强大的工具，但永远需要人来做最终判断。
特别是涉及安全、用户数据、生产环境的场景——**人必须兜底**。

</div>

# 当前设计好吗？有什么缺陷？

**这些是架构层面的限制，不是工程优化能解决的：**

**1. Transformer 的 O(n^2) 注意力机制**
> 上下文窗口越长，计算量平方级增长。128K 窗口已经很吃力，无限上下文目前不现实。

**2. 自回归生成（Autoregressive Generation）**
> 只能从左到右逐字生成，无法"先想好整体再落笔"。
> 就像写作文必须从第一个字开始写，不能先写结尾再补开头。

**3. 没有真正的推理能力**
> 本质上是超大规模的模式匹配，不是逻辑推演。
> 简单的数学题有时会算错，正是因为它在"猜"而非"算"。

**4. 训练数据截止**
> 模型的知识有截止日期，之后发生的事情它不知道。
> 需要 RAG（检索增强生成）来补充实时信息。

**5. 对齐（Alignment）仍然脆弱**
> 越狱攻击（Jailbreak）依然有效，安全护栏不是万无一失。

# AI + 业务系统：看得见的机会

<!-- _class: highlight -->

**技术团队可以落地的 AI 场景：**

| 场景 | 描述 | 落地难度 | 预期收益 |
|------|------|---------|---------|
| **智能客服** | 微调模型回答用户常见问题，减少人工客服压力 | 中 | 高 |
| **日志分析** | 实时分析系统日志，识别异常请求和潜在故障 | 高 | 极高 |
| **自动化运维** | 分析应用日志，预测服务故障、自动扩缩容 | 中 | 高 |
| **代码辅助** | 用 Cursor / CLI 提升开发团队效率 | 低 | 中高 |
| **文档智能** | 自动生成 / 翻译技术文档和用户手册 | 低 | 中 |

<div class="callout-blue">

**建议优先级：** 先从"代码辅助"和"文档智能"入手（成本低、见效快），
再逐步推进"智能客服"和"自动化运维"。

</div>

# 团队 AI 能力路线图

**三个月，从入门到业务落地：**

### 第 1 个月：工具上手

- 全员掌握 Cursor **高阶功能**（Agent Mode、@引用、.cursorrules）
- 全员体验 Claude CLI **基础操作**（对话、文件读写、代码搜索）
- 每人完成至少 **3 个 AI 辅助开发的真实任务**

### 第 2 个月：API 实践

- 团队能独立**调用 OpenAI / Claude API**
- 理解 Token、Temperature、System Prompt 等核心参数
- 构建至少 **1 个简单的 AI 功能原型**（如内部问答 Bot）

### 第 3 个月：业务落地

- 选定 **1-2 个业务场景**，开发 AI 解决方案原型
- 进行效果评估和成本分析
- 输出**可复制的实施方案**

> **持续机制：** AI Code Review 作为标准流程 + 每周知识分享会
