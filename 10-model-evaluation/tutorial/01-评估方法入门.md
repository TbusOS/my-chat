# 评估方法入门

## 为什么需要评估？

你微调了一个模型，跑了几个问题，"感觉不错"——这够吗？

**不够。** 原因如下：

| 问题 | 说明 |
|------|------|
| 样本偏差 | 你手动测试的几个例子可能恰好是模型擅长的 |
| 无法对比 | "感觉 A 比 B 好"在团队协作中毫无说服力 |
| 回归盲区 | 新版本在某些能力上可能悄悄退化了 |
| 成本决策 | 领导问"花这么多钱微调值不值"，你需要数据 |

科学的评估让你从"我觉得"升级到"数据显示"。

## 自动评估 vs 人工评估

### 自动评估

用程序自动运行测试集，计算量化指标。

```python
# 伪代码：自动评估流程
for question, reference_answer in test_dataset:
    model_answer = model.generate(question)
    score = compute_metric(model_answer, reference_answer)
    scores.append(score)

final_score = average(scores)
```

**优点**：可重复、成本低、速度快、可大规模运行

**缺点**：无法完全捕捉语言质量（流畅度、创造性等）

### 人工评估

由人类评估员对模型输出打分。

**优点**：能捕捉细微的语言质量差异

**缺点**：成本高、速度慢、评估员之间一致性难保证

### LLM-as-Judge（LLM 当裁判）

用更强的模型（如 GPT-4）评估较弱模型的输出。这是近年来流行的折中方案。

```python
judge_prompt = f"""请评估以下回答的质量（1-5分）：

问题：{question}
回答：{model_answer}

评分标准：
- 准确性（1-5）
- 完整性（1-5）
- 流畅度（1-5）
"""

score = judge_model.generate(judge_prompt)
```

**适用场景**：开放式生成任务、对话质量评估

## 常见评估指标

| 指标 | 全称 | 衡量什么 | 适用场景 | 值域 |
|------|------|----------|----------|------|
| Perplexity | 困惑度 | 模型对文本的"惊讶程度" | 语言建模 | 越低越好 |
| BLEU | Bilingual Evaluation Understudy | 生成文本与参考文本的 n-gram 重合度 | 翻译、文本生成 | 0-100 |
| ROUGE | Recall-Oriented Understudy for Gisting Evaluation | 生成摘要与参考摘要的重合度 | 文本摘要 | 0-1 |
| Pass@k | - | 生成 k 个代码方案中至少一个通过的概率 | 代码生成 | 0-1 |
| Accuracy | 准确率 | 回答正确的比例 | 选择题基准 | 0-1 |
| F1 Score | - | 精确率和召回率的调和平均 | 问答、信息抽取 | 0-1 |

## 如何选择评估方法？

根据你的场景选择：

| 你的场景 | 推荐评估方式 |
|----------|-------------|
| 选择基座模型 | 看排行榜 + 在你的数据上跑 Perplexity |
| 微调后验证 | 在固定测试集上比较微调前后的指标 |
| 中文能力评估 | C-Eval + CMMLU |
| 代码能力评估 | HumanEval + MBPP |
| 数学推理评估 | GSM8K + MATH |
| 对话质量评估 | LLM-as-Judge + 人工抽样 |
| 部署前最终验证 | 综合多个基准 + 业务测试集 |

## 关键原则

1. **用测试集评估，永远不要用训练集** -- 否则你测的是记忆力，不是能力
2. **固定评估条件** -- 相同的 prompt 模板、温度参数、最大长度
3. **多维度评估** -- 单一指标无法全面反映模型能力
4. **关注你的场景** -- 排行榜第一不代表在你的任务上最好
